# Build offline PWA for forage

## Design

### What it is

A Progressive Web App generated by `forage export --pwa -o ./pwa/`, producing a directory of static files (HTML, JS, CSS, manifest, service worker, icons). Installable on mobile, works offline, supports full CRUD on books with manual diff-based export for syncing back to the CLI.

### Data layer

- **Initial load:** Book and bookseller data embedded as JSON in the HTML (same as today's export)
- **Persistence:** Written to IndexedDB on first load. Subsequent visits read from IndexedDB.
- **Change tracking:** Every mutation (create/update/delete) is logged to a `changes` store in IndexedDB with timestamp and operation type
- **Export to CLI:** "Download Changes" exports the changelog as JSON. CLI applies the operations via `forage import --changes`
- **Re-import from CLI:** Running `forage export --pwa` again produces fresh embedded data. PWA detects newer data and offers to replace, clearing the changelog.

### CLI changes

- Add `--pwa` flag to `forage export`
- When set, output a directory instead of a single HTML file:
  ```
  pwa/
    index.html        # app shell + embedded data
    app.js            # all app logic
    style.css         # styles
    manifest.json     # PWA manifest
    sw.js             # service worker (cache-first)
    icon-192.png      # generated or placeholder
    icon-512.png
  ```
- Add `forage import --changes` to apply PWA changelog diffs to the master DB

### PWA features

- **Installable:** manifest.json with app name, icons, theme color
- **Offline:** Service worker caches all assets on first visit
- **Full CRUD:** Add books, edit all fields (title, author, status, rating, tags, notes), delete books
- **Open Library lookup:** Add-book form has a "Look up" button that queries openlibrary.org/search.json directly (CORS-friendly, no key). Shows canonical title, author, first publish year. User confirms before saving.
- **UI:** Evolve the current card layout — same warm palette, add edit modals/inline editing, mobile-optimized touch targets
- **Search/filter/sort:** Same as current export
- **Sync:** "Download Changes" button exports changelog as JSON diffs

### What's NOT included (YAGNI)

- No server component
- No automatic sync
- No SQLite WASM
- No user accounts or auth
- No book cover images

## Implementation Plan

**Goal:** Build an installable, offline-capable PWA that provides full CRUD on the forage book library with diff-based sync back to the CLI.

**Architecture:** The Go CLI embeds static PWA assets via `go:embed` and writes them to a directory when `forage export --pwa` is used. Book/bookseller data is injected into `index.html` as JSON (same pattern as current HTML export). The PWA uses IndexedDB for persistence and tracks all mutations in a changelog store. A new `forage import --changes` command applies changelog JSON to the master SQLite DB.

**Tech Stack:** Go (`embed`, `html/template`, `encoding/json`), vanilla JS (no framework), IndexedDB, Service Worker API, Web App Manifest.

### Task 1: PWA export infrastructure

**Files:**
- Create: `internal/pwa/pwa.go`
- Create: `internal/pwa/pwa_test.go`
- Modify: `cmd/export.go` (add `--pwa` flag, directory output path)

**Behavior:**
Add a `--pwa` flag to `forage export`. When set, create the output directory and write all PWA files into it. The `pwa.Generate()` function takes books, booksellers, and an output directory path. It writes `index.html` (with data injected), plus static files (`app.js`, `style.css`, `manifest.json`, `sw.js`). Static files are embedded with `go:embed` and written as-is. `index.html` uses `html/template` to inject the JSON data (same pattern as `internal/export/html.go`).

**Testing:**

- `TestGenerate_EmptyLibrary` — verify it works with zero books

**Notes:**
The `index.html` template is minimal in this task — just an app shell with embedded data. The full UI comes in later tasks. Start with a placeholder that proves the pipeline works.

### Task 2: IndexedDB data layer (`app.js` — storage module)

**Files:**
- Create: `internal/pwa/assets/app.js` (this grows across tasks; start with the data layer)

**Behavior:**
JavaScript module that manages IndexedDB. Two object stores: `books` (keyed by `id`) and `changes` (auto-increment key). On first load, reads embedded JSON from `window.__FORAGE_DATA__` and populates IndexedDB. On subsequent loads, reads from IndexedDB directly. Detects "newer embedded data" by comparing a `dataVersion` timestamp embedded in the HTML against one stored in IndexedDB.

API:
- `db.init()` — open DB, seed from embedded data if needed
- `db.listBooks()` — return all books
- `db.getBook(id)` — return one book
- `db.createBook(book)` — insert + log change `{op:"create", book, ts}`
- `db.updateBook(id, fields)` — update + log change `{op:"update", id, fields, ts}`
- `db.deleteBook(id)` — delete + log change `{op:"delete", id, ts}`
- `db.getChanges()` — return all changelog entries
- `db.clearChanges()` — clear changelog after export

ID generation for new books: 4-char hex from `crypto.getRandomValues()`, same format as CLI.

**Testing:**
No automated JS tests — this is vanilla JS with no build tooling. Verified manually and through the integration of later tasks. The Go-side embed tests from Task 1 cover that the file is correctly written.

**Notes:**
Use an IIFE or module pattern. No imports, no build step. The `window.__FORAGE_DATA__` global is set by a `<script>` tag in `index.html` before `app.js` loads.

### Task 4: Full CRUD UI

**Files:**
- Modify: `internal/pwa/assets/app.js` (add modal/form logic)
- Modify: `internal/pwa/assets/style.css` (modal styles)
- Modify: `internal/pwa/assets/index.html` (modal markup)

**Behavior:**
Three interactions:

**Add book:** FAB button (bottom-right) opens a modal form with fields: title, author, status (dropdown), rating (1-5 or none), tags (comma-separated text input), notes (textarea). "Look up" button next to title (Task 5). Save calls `db.createBook()`, closes modal, re-renders list.

**Edit book:** Tap a book card to open the same modal pre-filled with current values. Save calls `db.updateBook()`. Cancel closes without saving.

**Delete book:** Delete button in edit modal. Confirmation prompt. Calls `db.deleteBook()`.

Validation: title and author required. Status must be one of: wishlist, reading, paused, read, dropped. Rating 0-5.

**Testing:**

**Notes:**
Depends on Task 2 (data layer) and Task 3 (list rendering). The modal should be a single reusable element toggled between add/edit modes.

### Task 5: Open Library lookup in add form

**Files:**
- Modify: `internal/pwa/assets/app.js` (add OL search function and UI wiring)

**Behavior:**
"Look up" button in the add-book modal. When clicked, calls `https://openlibrary.org/search.json?title=...&author=...&limit=5&fields=title,author_name,first_publish_year`. Shows results in a dropdown/list below the button. User taps a result to fill in the form fields (title, author). If no results, show "No matches found."

Only available during add, not edit. Requires network — gracefully show "Lookup unavailable offline" if fetch fails.

**Testing:**
- Manual: add a book, click "Look up", verify results appear, select one, verify form fills in.

**Notes:**
Uses `fetch()` directly. The Open Library search API is CORS-friendly and requires no key. Same endpoint and fields as the CLI's `internal/openlibrary` package. `limit=5` instead of `limit=1` to let the user pick the right match.

### Task 6: Service worker + manifest

**Files:**
- Create: `internal/pwa/assets/sw.js`
- Create: `internal/pwa/assets/manifest.json`
- Modify: `internal/pwa/assets/index.html` (register SW, link manifest)

**Behavior:**
**Service worker:** Cache-first strategy. On install, pre-cache all app assets (`index.html`, `app.js`, `style.css`, `manifest.json`, icons). On fetch, serve from cache, fall back to network. Version the cache name so re-exports bust the cache.

**Manifest:** App name "Forage", short name "Forage", `display: standalone`, theme color `#f8f7f4`, background color `#f8f7f4`. Icons at 192px and 512px (simple placeholder SVG-based PNGs or inline SVG icons).

**index.html:** Register the service worker. Add `<link rel="manifest">` and `<meta name="theme-color">`.

**Testing:**
- Go test: `TestGenerate_ManifestValid` — verify manifest.json is valid JSON with required fields
- Go test: `TestGenerate_ServiceWorkerExists` — verify sw.js is written
- Manual: open in Chrome, verify "Install app" prompt appears, verify offline mode works

### Task 7: Download changes (export changelog)

**Files:**
- Modify: `internal/pwa/assets/app.js` (add export UI and logic)
- Modify: `internal/pwa/assets/style.css` (sync button styles)

**Behavior:**
"Download Changes" button in a toolbar/header area. Shows a badge with the number of pending changes. When clicked:
1. Calls `db.getChanges()` to get all changelog entries
2. Creates a JSON blob with format:
   ```json
   {
     "version": 1,
     "exported": "2026-03-01T12:00:00Z",
     "changes": [
       {"op": "create", "book": {...}, "ts": "..."},
       {"op": "update", "id": "f10d", "fields": {"status": "read", "rating": 4}, "ts": "..."},
       {"op": "delete", "id": "a3b2", "ts": "..."}
     ]
   }
   ```
3. Triggers a file download (`forage-changes.json`)
4. Asks "Clear changelog?" — if yes, calls `db.clearChanges()`

If no changes, button shows "No changes" and is disabled.

**Testing:**
- Manual: make some edits, download changes, verify JSON structure is correct.

### Task 8: CLI import --changes

**Files:**
- Modify: `cmd/import.go` (add `--changes` flag and JSON changelog handler)
- Create: `cmd/import_changes_test.go`

**Behavior:**
`forage import --changes forage-changes.json` reads the changelog JSON and applies operations in order:
- `create`: call `store.CreateBook()` with the book data. Use the PWA-generated ID if it doesn't collide; otherwise generate a new one.
- `update`: call `store.UpdateBook()` for each field in the change.
- `delete`: call `store.DeleteBook()`.

Output:
```json
{"applied": 5, "skipped": 0, "errors": 0}
```

Skip changes that can't be applied (e.g., updating a deleted book) with a warning to stderr. Don't fail the whole import on individual errors.

**Testing:**
- `TestImportChanges_Create` — apply a create change, verify book exists in DB
- `TestImportChanges_Update` — create a book, apply an update change, verify fields changed
- `TestImportChanges_Delete` — create a book, apply a delete change, verify book is gone
- `TestImportChanges_SkipMissing` — apply an update to a non-existent book, verify it's skipped without error
- `TestImportChanges_MixedOps` — apply a mix of creates, updates, and deletes in one changelog

**Notes:**
The `--changes` flag is mutually exclusive with the existing CSV import behavior. When `--changes` is set, the positional arg is treated as a JSON changelog file, not a CSV.
